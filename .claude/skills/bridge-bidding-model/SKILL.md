# Bridge bidding model

A bidding *system* is a tree of *calls*. The tree is stored as JSONB in
`bidding_system.tree_json` and is treated **opaquely by the backend** — all
domain logic lives in the frontend, primarily [tree.ts](../../../frontend/src/tree.ts).

## Node shape

```ts
interface BidNode {
  id: string;          // unique; generated by newId() = `node-${crypto.randomUUID()}`
  bids: string[];      // ≥1 calls; [] only for the synthetic root sentinel
  meaning: string;     // brief one-liner
  notes?: string;      // optional long-form explanation (multi-paragraph)
  byOpponent?: boolean;// interference / opponent call
  children: BidNode[];
}
```

A node holds **one or more equivalent calls** that share the same meaning and
continuations (a "group"). The synthetic root has `bids: []`; everything else
has `bids.length ≥ 1`.

> Legacy migration: older data used a single `bid: string` field. `rootFromTree`
> in [tree.ts](../../../frontend/src/tree.ts) migrates `bid` → `bids: [bid]` on
> load. Never reintroduce a singular `bid` field.

## Call notation

A call string is one of:

| Form | Meaning | Examples |
|---|---|---|
| `<level><strain>` | A contract bid | `1♣`, `2NT`, `7♠` |
| `X` | Double | — |
| `XX` | Redouble | — |

- **Levels**: `1`–`7`
- **Strains** (display order): `♣ ♦ ♥ ♠ NT ma om`
  - `♣ ♦ ♥ ♠ NT` are literal
  - `ma` = **same major** (the major just bid in this auction)
  - `om` = **other major** (the major not bid)
  - `ma`/`om` are symbolic placeholders, useful in continuations under grouped
    major openings (e.g. under `1♥/1♠`, `2ma` = "raise opener's major").
- `X`/`XX` are always **single-call** — never grouped, never combined with a contract bid in the same node.

## Ordering rules

Two ranks exist and must not be confused:

1. **Display rank** (`STRAIN_RANK`): drives picker layout. `ma`=5, `om`=6 (after NT).
2. **Comparison rank** (`strainComparisonRank`, private to tree.ts): drives "is
   this bid higher". Both `ma` and `om` compare as `♠` (the higher major) — the
   most permissive interpretation. So `2♥ < 2ma`, `2♠ ≮ 2ma`, `2ma < 2NT`.

`bidRank` uses the **comparison** rank. When adding strain comparisons, use
`strainOutranks(s, parent)` or `compareBids` — never compare `STRAIN_RANK`
values directly (that would mis-rank `ma`/`om`).

## Chain context

Validity of a new call depends on the *chain* of ancestors, not just the
immediate parent. `chainContextFromNodes(nodes)` returns:

```ts
interface ChainContext {
  lastContractBid: string | null; // highest contract bid seen; the "bar"
  hasActiveDouble: boolean;       // last contract is doubled, not yet redoubled
  hasActiveRedouble: boolean;     // last contract is doubled AND redoubled
}
```

- A new contract bid **resets** the double/redouble state.
- For a **group** of contract bids, the *highest* in the group sets `lastContractBid`
  (`highestContractBidIn`). So under `1♥/1♠`, the bar for the next contract is `1♠`.
- `addChainContext(root, parentId)` — context for adding *under* a node (includes it).
- `editChainContext(root, nodeId)` — context for editing a node (excludes it).

## Legality

| Call | Legal when |
|---|---|
| Contract bid | Strictly higher than `lastContractBid` (`isValidContinuation`) |
| Double (`X`) | `lastContractBid` exists AND `!hasActiveDouble` AND `!hasActiveRedouble` |
| Redouble (`XX`) | `hasActiveDouble` AND `!hasActiveRedouble` |
| Opening (root) | Any contract bid; X/XX disabled (no contract to act on) |

The bid form ([BidForm.tsx](../../../frontend/src/components/BidForm.tsx))
enforces all of this by enabling/disabling level + strain pills and the
Bid/Double/Redouble segmented control, with tooltips explaining why a choice
is disabled.

## Rendering

All call rendering goes through [BidLabel.tsx](../../../frontend/src/components/BidLabel.tsx):
- Single bid → one colored token.
- Group → each call in its own suit color, separated by a subtle `/`.
- Interference (`byOpponent`) → whole label wrapped in `(…)`, dimmed + italic.

Colors come from `suitColor`/`suitOf` in [styles.ts](../../../frontend/src/styles.ts):
`♥ ♦`=red, `♠ ♣`=black, `NT`=blue, `ma om`=purple (`--suit-symbolic`),
`X`=red, `XX`=blue.

## Common pitfalls

- **Never** generate node IDs with a module-level counter — it collides across
  reloads. Always use `newId()`.
- The backend never inspects bids. Don't add bid logic server-side; it would
  duplicate (and drift from) the frontend rules.
- When adding a new strain or call kind, update: `STRAINS`, `STRAIN_RANK`,
  `strainComparisonRank`, `parseBid`, `isContractBid`, `suitOf`, `suitColor`,
  and the BidForm pickers.
